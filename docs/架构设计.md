# 通用智能客服平台 — 架构设计

> 目标：构建一个**可复用、可商业化**的智能客服平台。  
> 任何垂直行业（八字命理、法律咨询、医疗问诊、电商售后……）只需接入"领域插件"，即可拥有完整的智能客服能力。  
> 将八字智能客服作为**第一个领域实例**，验证平台可行性。

---

## 一、商业模式

### 1.1 三种变现路径

| 模式 | 说明 | 目标客户 | 收入模型 |
|------|------|----------|----------|
| **SaaS 平台** | 客户自助开通，配置领域知识即可上线 | 中小企业、个人IP | 按月订阅 + 按对话量阶梯计费 |
| **PaaS 能力输出** | 提供 API / SDK，客户集成到自有系统 | 中大型企业、ISV | API 调用量计费 |
| **垂直解决方案** | 针对特定行业（命理、法律、医疗）交付整套方案 | 行业客户 | 项目制 + 年费维护 |

### 1.2 定价策略（SaaS 参考）

| 套餐 | 月费 | 对话量 | 坐席数 | 特性 |
|------|------|--------|--------|------|
| **免费版** | ¥0 | 100次/月 | 1 | 基础问答、单渠道 |
| **专业版** | ¥299/月 | 5000次/月 | 5 | 多渠道、记忆体系、数据分析 |
| **企业版** | ¥999/月 | 无限 | 无限 | 私有部署、SLA保障、专属模型 |
| **按量付费** | — | ¥0.05/次 | — | 超出套餐后按量计费 |

## 三、核心模块详解

### 3.1 多租户系统 (Tenant)

商业化的基石。每个"客户"就是一个租户。

```
tenant/
├── tenant_manager.py        # 租户 CRUD
├── tenant_config.py         # 租户配置（模型、额度、域名）
├── billing.py               # 计费引擎
├── quota.py                 # 额度管理 & 限流
└── models.py                # 数据模型
```

**关键设计：**
- **数据隔离**：Schema 级隔离（PostgreSQL Schema）或行级隔离（tenant_id 字段）
- **配置隔离**：每个租户可独立配置模型、提示词、知识库、UI 主题
- **计费精确**：每次对话、每次 Token 消耗都归属到具体租户
- **弹性额度**：支持预付费（包月）和后付费（按量）混合模式

```python
# 租户配置示例
class TenantConfig:
    tenant_id: str
    name: str
    plan: str                    # free | pro | enterprise
    domain_plugins: list[str]    # ["bazi", "custom_qa"]
    llm_config: dict             # 模型选择、参数
    branding: dict               # Logo、颜色、欢迎语
    quota: QuotaConfig           # 对话额度
    webhook_url: str             # 事件回调
```

### 3.2 对话引擎 (Conversation Engine)

平台的心脏。管理完整的对话生命周期。

```
conversation/
├── engine.py                # 对话主循环
├── context.py               # 上下文构建器
├── router.py                # 意图 → 领域路由
├── state_machine.py         # 会话状态机
├── stream.py                # 流式输出管理
└── handoff.py               # 人机转接
```

**对话流程：**

```
用户消息 → 预处理(敏感词/注入检测)
        → 意图识别(LLM/规则混合)
        → 领域路由(分发到对应插件)
        → 上下文构建(记忆 + RAG + 槽位)
        → LLM 编排(Prompt组装 + 模型调用)
        → 后处理(格式化/敏感信息过滤)
        → 流式返回 + 异步存储
```

**状态机设计：**

```
[初始] → [意图收集] → [槽位填充] → [执行中] → [结果呈现]
  ↓         ↓             ↓            ↓           ↓
  └─────────┴─────────────┴────────────┴───→ [人工接管]
                                                   ↓
                                             [会话结束]
```

### 3.3 领域插件系统 (Domain Plugin)

**这是平台通用化的核心。** 每个行业/场景就是一个插件。

```
domains/
├── base_plugin.py           # 插件基类（接口定义）
├── plugin_loader.py         # 插件加载器
├── plugin_registry.py       # 插件注册表
│
├── bazi/                    # 八字命理插件（第一个实例）
│   ├── plugin.py            # 插件入口
│   ├── engine.py            # 排盘引擎
│   ├── prompts/             # 提示词模板
│   ├── knowledge/           # 知识库文件
│   └── config.yaml          # 插件配置
│
├── legal/                   # 法律咨询插件（示例）
│   ├── plugin.py
│   ├── ...
│
└── template/                # 插件模板（快速创建新插件）
    ├── plugin.py
    ├── prompts/
    ├── knowledge/
    └── config.yaml
```

**插件接口定义：**

```python
class BaseDomainPlugin(ABC):
    """领域插件基类，所有行业插件必须实现这些接口"""
    
    @abstractmethod
    def get_intents(self) -> list[Intent]:
        """声明本插件处理的意图列表"""
        pass
    
    @abstractmethod
    def get_slots(self, intent: str) -> list[Slot]:
        """声明完成某意图需要的槽位"""
        pass
    
    @abstractmethod
    def build_context(self, session: Session) -> str:
        """构建领域特定的上下文（注入到 Prompt）"""
        pass
    
    @abstractmethod
    def get_system_prompt(self, intent: str) -> str:
        """返回领域 System Prompt"""
        pass
    
    @abstractmethod
    def post_process(self, response: str, session: Session) -> str:
        """领域特定的后处理（格式化、补充信息等）"""
        pass
    
    def get_tools(self) -> list[Tool]:
        """可选：声明领域工具（Function Calling）"""
        return []
    
    def on_session_start(self, session: Session):
        """可选：会话开始时的初始化逻辑"""
        pass
    
    def on_session_end(self, session: Session):
        """可选：会话结束时的清理/总结逻辑"""
        pass
```

### 3.4 记忆系统 (Memory)

沿用八字项目的三层架构，升级为**多租户通用版**。

```
memory/
├── manager.py               # 统一记忆管理器
├── short_term.py            # 短期记忆（Redis，会话级）
├── user_profile.py          # 用户画像（PostgreSQL，跨会话）
├── long_term.py             # 长期情景记忆（向量数据库）
└── consolidation.py         # 记忆整合（定期从短期→长期）
```

**记忆调用链路：**

```
新消息到达
  → 加载短期记忆（本次会话历史，Redis）
  → 加载用户画像（历史偏好/标签，PostgreSQL）
  → 语义检索长期记忆（相关历史片段，向量DB）
  → 组装成 context 注入 Prompt
  → LLM 生成回复
  → 异步更新记忆（短期追加 + 画像更新 + 长期入库）
```

**商业价值点：**
- 用户画像标签体系 → **精准营销** 的数据基础
- 长期记忆 → **个性化服务** 的核心壁垒（用户迁移成本高）
- 对话分析 → **需求洞察** 和 **产品优化** 的数据来源

### 3.5 RAG 检索增强 (Retrieval)

```
rag/
├── indexer.py               # 文档索引（分块 + 向量化）
├── retriever.py             # 检索器（语义 + 关键词混合）
├── reranker.py              # 重排序（交叉编码器）
├── knowledge_base.py        # 知识库管理（CRUD）
└── chunking.py              # 智能分块策略
```

**知识库管理（面向租户）：**
- 支持上传文档（PDF/Word/Markdown/TXT）
- 自动分块 → 向量化 → 入库
- 后台管理界面：预览、编辑、删除知识条目
- 支持手动 Q&A 对添加（高优先级匹配）

### 3.6 LLM 编排层 (LLM Router)

```
llm/
├── router.py                # 模型路由（按租户/场景选模型）
├── providers/               # 模型供应商适配
│   ├── openai.py
│   ├── deepseek.py
│   ├── qwen.py
│   ├── zhipu.py
│   └── local.py             # 本地模型（Ollama）
├── prompt_builder.py        # Prompt 组装器
├── token_counter.py         # Token 计数 & 成本核算
└── fallback.py              # 降级策略（主模型不可用时切备用）
```

**模型路由策略：**

| 场景 | 推荐模型 | 原因 |
|------|----------|------|
| 简单FAQ | Qwen-turbo / GPT-3.5 | 成本低、速度快 |
| 复杂推理 | DeepSeek-V3 / GPT-4o | 准确度高 |
| 领域专精 | 微调模型 / DeepSeek | 领域表现好 |
| 敏感场景 | 本地 Ollama | 数据不出域 |

**成本优化：**
- 简单问题走小模型（¥0.001/次），复杂问题走大模型（¥0.01/次）
- 通过意图识别自动路由，平均成本降低 60-70%

### 3.7 人机协作 (Human Handoff)

```
handoff/
├── detector.py              # 转人工检测（置信度/情绪/关键词）
├── queue.py                 # 排队管理
├── agent_panel.py           # 人工坐席后台
├── copilot.py               # AI 辅助坐席（推荐回复）
└── takeover.py              # 接管/归还逻辑
```

**转人工触发条件：**
- 用户主动要求（"转人工"）
- AI 置信度连续 3 轮低于阈值
- 检测到负面情绪升级
- 涉及敏感话题（投诉、退款、法律风险）
- 租户自定义规则

**AI Copilot 模式：**
- 人工坐席接管后，AI 退到后台
- 实时推荐回复（坐席一键采纳或修改）
- 自动调取知识库、用户画像供坐席参考
- 会话结束后 AI 自动生成摘要

### 3.8 数据分析 (Analytics)

```
analytics/
├── collector.py             # 数据采集（埋点）
├── dashboard.py             # 数据看板 API
├── reports.py               # 报表生成
└── insights.py              # 智能洞察
```

**核心指标（租户看板）：**

| 类别 | 指标 |
|------|------|
| **效率** | 平均响应时间、首次解决率、会话时长 |
| **质量** | 用户满意度(CSAT)、转人工率、AI解决率 |
| **成本** | 单次对话成本、Token消耗、人工坐席成本节省 |
| **增长** | DAU/MAU、会话量趋势、新用户转化率 |
| **业务** | 高频问题 TOP10、未覆盖意图、知识库命中率 |

## 四、技术选型

### 4.1 后端技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| **语言** | Python 3.12+ | 生态最好，AI库最丰富 |
| **Web框架** | FastAPI | 异步、高性能、自动文档 |
| **ORM** | SQLAlchemy 2.0 | 成熟稳定，多数据库支持 |
| **任务队列** | Celery + Redis | 异步任务（记忆整合、报表生成） |
| **缓存** | Redis | 会话状态、限流、排队 |
| **关系数据库** | PostgreSQL | 多租户、JSON支持、性能强 |
| **向量数据库** | ChromaDB → Milvus | 初期ChromaDB够用，规模化换Milvus |
| **对象存储** | MinIO / OSS | 文档、图片、日志 |
| **消息队列** | Redis Streams / RabbitMQ | 事件驱动、解耦 |

### 4.3 部署 & 运维
