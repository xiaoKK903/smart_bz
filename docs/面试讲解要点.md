# 面试讲解要点

> 面试时按这个结构讲，3-5 分钟讲清楚，然后等面试官追问。

---

## 一句话介绍

> "我设计并实现了一个**插件化智能客服平台**，通过领域插件机制让不同行业快速接入。目前落地了两个场景：**命理咨询**和**电商售后**，验证了架构的通用性。"

---

## 30 秒版（电梯演讲）

"传统做法是每个行业写一套客服系统。我的方案是把**对话引擎、记忆系统、RAG检索、LLM编排**抽成平台层，行业差异全封装在插件里——一个 YAML 配置 + 几个 Python 接口就能上线一个新场景。

第一个插件是命理咨询，验证了多轮槽位填充和个性化记忆；第二个是电商售后，验证了外部 API 集成、人机转接和情绪检测。两个场景的底层引擎完全复用，零修改。"

---

## 双插件对照表（面试核心弹药）

| 维度 | 八字命理 | 电商售后 | 说明什么 |
|------|---------|---------|---------|
| **意图数** | 3-5 个（排盘、运势、关系） | 15+ 个（订单/物流/退款/投诉…） | 意图引擎能处理简单→复杂 |
| **槽位类型** | 日期、时辰、性别 | 订单号、手机号、金额、图片 | 槽位系统支持多种类型 |
| **外部集成** | 无（纯本地计算） | 订单API、物流API、支付API | 插件可自由接外部系统 |
| **知识库** | 命理典籍（固定知识） | 退换货政策 + 商品库（动态更新） | RAG 应对不同知识形态 |
| **记忆重点** | 用户命盘、过往解读 | 订单历史、投诉记录、VIP等级 | 记忆系统可存不同领域数据 |
| **人机转接** | 几乎不需要 | 投诉/复杂场景必须转 | 转接规则可插件级配置 |
| **模型温度** | 0.7（需要生动解读） | 0.3（需要准确稳定） | 模型参数按场景调优 |
| **后处理** | 排版美化 | 追加操作链接/按钮 | 后处理逻辑按场景定制 |
| **确认机制** | 不需要 | 退款操作需用户确认 | 插件声明是否需要确认 |
| **Tool Calling** | 无 | 查订单、查物流、提交退款 | Function Calling 按需启用 |

**面试话术：**
> "您可以看到，这两个场景在业务上完全不同，但底层的对话引擎、记忆系统、RAG模块、流式输出全部复用。这就是插件架构的价值——**平台稳定，插件灵活**。"

## 预判面试官会问的问题 & 准备

### Q1: "多租户数据怎么隔离的？"

**回答：**
"两种方案，我选了**行级隔离**（每条记录带 `tenant_id`），原因是初期租户少，Schema 级隔离运维成本高。查询层统一加 `WHERE tenant_id = ?`，通过中间件自动注入，业务代码不需要感知。

如果到了企业版客户需要物理隔离，可以升级到**独立数据库实例**，路由层按租户切数据源就行。"

### Q2: "意图识别怎么做的？准确率多少？"

**回答：**
"混合方案——**规则优先 + LLM 兜底**。
- 高频明确意图（'查物流'、'退款'）用正则规则匹配，命中率 90%+，零 Token 成本
- 模糊/复杂意图走 LLM 分类，用 few-shot prompting，准确率约 85-90%

这样做的好处：80% 的请求被规则拦截，LLM 只处理 20% 的长尾，**Token 成本降了 4 倍**。"

### Q3: "记忆体系为什么要三层？一层不行吗？"

**回答：**
"解决三个不同的问题：
- **短期记忆**（Redis）：当前会话上下文，TTL 30 分钟。解决'上文说过的事不用重复问'
- **用户画像**（PostgreSQL）：结构化用户数据，跨会话持久化。解决'老用户不用重新介绍自己'
- **长期情景记忆**（向量数据库）：历史对话的语义检索。解决'三个月前聊的相关话题能想起来'

单层记忆要么太短（丢上下文），要么太贵（所有历史都塞 Prompt），三层各取所需。"

### Q4: "模型路由是怎么实现的？"

**回答：**
"按**意图复杂度**路由：
- 简单 FAQ / 槽位确认 → 小模型（Qwen-turbo），成本 ¥0.001/次
- 复杂推理 / 情感安抚 / 投诉处理 → 强模型（DeepSeek-V3），成本 ¥0.01/次

意图识别阶段就确定了复杂度，路由发生在 LLM 调用前。平均下来比全用大模型**省 60-70% 成本**。

还有降级机制——如果主模型超时或报错，自动 fallback 到备用模型，保证可用性。"

### Q5: "人机转接的判断标准是什么？"

**回答：**
"四种触发条件，可以在插件配置里自定义权重：
1. **显式请求**：用户说'转人工'，立即转
2. **低置信度**：连续 3 轮 AI 置信度 < 0.4，可能在答非所问
3. **情绪检测**：用户情绪评分持续恶化（情感分析），需要人来安抚
4. **死循环检测**：超过 N 轮未解决，说明 AI 处理不了

不同插件阈值不同——八字场景几乎不转，电商投诉场景 2 轮就转。这个可配置。"

### Q6: "RAG 检索的效果怎么保证？"

**回答：**
"三个手段：
1. **混合检索**：语义向量搜索 + BM25 关键词搜索，取并集再 rerank
2. **精确 QA 对优先**：高频问题直接走精确匹配，不走向量
3. **分块策略**：按语义段落分块（不是固定 500 字），保持上下文完整

知识库后台可以看到每次检索命中了哪些片段、分数多少，方便调优。"

### Q7: "这个项目的难点是什么？"

**可选回答（挑你最有感触的）：**

**技术难点：**
> "插件接口设计。太抽象了插件写起来费劲，太具体了新场景接不进来。我迭代了三版才找到平衡点——5 个必须接口 + 3 个可选接口，核心靠 YAML 配置驱动，减少代码量。"

**工程难点：**
> "记忆系统的一致性。短期记忆在 Redis、画像在 PostgreSQL、长期记忆在向量数据库，三个存储的数据要保持一致，特别是用户删除记忆的时候需要三个都删。我做了一致性检查 API 和定期对账任务来兜底。"

**业务难点：**
> "提示词调优。同一个插件，不同意图需要不同的 System Prompt 和温度参数。电商投诉场景温度太高回复不稳定，太低又太冷冰冰。最后按意图级别做了参数覆盖。"

### Q8: "为什么不用 LangChain？"

**回答：**
> "初版有意不用，原因是 LangChain 的抽象层太重，debug 困难，而且客服场景的链路比较确定（不需要 Agent 自主决策），用不上它最核心的能力。
>
> 不过架构上预留了扩展点——如果后续需要复杂的多步任务编排（比如：查订单→判断状态→调物流→算退款金额→提交），可以引入 LangGraph 做状态机编排，不影响现有代码。"

## 架构图白板画法（面试现场）

面试官给你白板的时候，画这个简化版：

```
用户 → [Widget/微信/API]
            ↓
      [API Gateway] ← 认证/限流/租户路由
            ↓
      [对话引擎]
       ↙   ↓   ↘
   意图   记忆   RAG
   识别   系统   检索
       ↘   ↓   ↙
      [Prompt 组装]
            ↓
      [LLM Router] → 小模型/大模型/本地模型
            ↓
      [领域插件] ← 八字 | 电商 | 法律 | ...
       ↙        ↘
  直接回复    调用外部API（订单/物流）
            ↓
      [流式返回 + 异步存储]
```

画完指着"领域插件"那一层说：**"这层是可插拔的，上面的全复用。"** 面试官会懂。
